# CLAUDE.md

> Claude Code 작업 규칙. 모든 세션 시작 시 이 문서를 먼저 읽고 작업한다.

---

## 1. 이 프로젝트가 무엇인가 (한 줄)

메인프레임 → 리눅스 마이그레이션의 **배치 출력 동등성을 자동 검증**하는 도구의 **Working 프로토타입**. 외주 납품용. E2E 시연 가능해야 함.

> 자세한 사업·도메인 맥락은 반드시 `CONTEXT.md`를 먼저 읽을 것.

---

## 2. 작업 시작 전 필수 읽기 순서

세션을 시작할 때, 또는 새로운 Task에 들어갈 때 반드시 아래 순서대로 컨텍스트를 로드한다.

1. **`CONTEXT.md`** — 사업 맥락, 왜 만드는지, 누구를 위한 것인지
2. **`ARCHITECTURE.md`** — Core/CLI 분리 구조, 모듈 책임
3. **`SPEC.md`** — 기능 명세, 입출력 계약
4. **`TASKS.md`** — 작업 분할, 의존 관계, 현재 어느 Task인지
5. (필요 시) `DECISIONS.md` — 과거에 결정된 사항과 이유

> 컨텍스트가 부족하다고 느끼면 **추측하지 말고 문서로 돌아가서 읽거나, 사용자에게 질문할 것**.

---

## 3. 핵심 설계 원칙 (절대 어기지 말 것)

### 3-1. Core / Interface 분리 (가장 중요)

- **Core Layer**: 비교 엔진, 배치 호출, 리포트 생성 등 비즈니스 로직.
- **Interface Layer**: CLI(현재), GUI(추후 추가 예정).
- **Core는 인터페이스에 의존하지 않는다.** Core 함수는 `print()`로 결과를 출력하지 않고, **구조화된 객체(dataclass/dict)를 반환**한다. 출력은 Interface가 한다.
- **이유**: 향후 GUI를 같은 Core 위에 얹을 것이기 때문. CLI에 로직을 우겨넣지 말 것.

### 3-2. 결과는 항상 구조화된 데이터로

- 비교 결과는 `ComparisonResult` 같은 dataclass로 표현.
- CLI는 그 객체를 받아 사람이 보기 좋게 출력만 함.
- 리포트 CSV도 같은 객체에서 파생.

### 3-3. 설정은 코드 바깥으로

- 인코딩, OS, 경로, DB 접속 정보 등 **클라이언트마다 다를 수 있는 값**은 코드에 박지 않는다.
- 설정 파일(YAML 또는 TOML)로 분리.
- 기본값(default)은 우분투 + Shift-JIS.

### 3-4. 결정론적 비교 (LLM 금지)

- 비교 판정은 **반드시 결정론적 코드**로. LLM/AI 호출 금지.
- 검증 도구의 생명은 신뢰성·재현성.

### 3-5. 외부 의존을 막 키우지 말 것

- 표준 라이브러리 우선. 꼭 필요한 패키지만.
- 무거운 프레임워크(Django 등) 도입 금지.
- 프로토 단계임을 잊지 말 것.

---

## 4. 작업 단위 규칙

### 4-1. Task 단위로 작업

- 한 번에 한 Task만 작업한다. `TASKS.md`에 정의된 단위 그대로.
- 여러 Task를 동시에 손대지 말 것. 컨텍스트가 흐려지고 품질이 떨어진다.

### 4-2. 각 Task 완료 시 체크

- 해당 Task의 "완료 기준(Definition of Done)"을 모두 만족하는지 자체 점검.
- 테스트가 정의되어 있으면 통과 확인.
- 완료되면 `TASKS.md`의 해당 항목을 `[x]`로 표시.

### 4-3. 새로운 결정 사항이 생기면 문서화

- 코드 작성 중 새로운 설계 결정을 내렸다면 `DECISIONS.md`에 한 줄 추가.
- 사용자와의 대화에서 새로운 합의가 나왔다면 마찬가지.
- **문서화되지 않은 결정은 다음 세션에서 잊혀진다.**

## 5. 코드 스타일

- **언어**: Python 3.10+
- **포맷터**: `black` 기본 설정 (line length 100)
- **타입 힌트**: 모든 public 함수에 필수
- **docstring**: 모든 public 함수에 한국어로 1~3줄
- **테스트**: `pytest`. 최소한 핵심 비교 로직과 적재 로직에 대해.
- **로그**: Core 안에서 `print()`는 절대 금지, `logging`은 OK (디버그·진단용). 사람이 보는 *최종 출력 포맷팅*은 Interface(CLI)의 책임이며 Core에 두지 않는다.

---

## 6. 금지 사항

- ❌ Core 로직 안에서 `print()` 호출 (CLI/GUI에서만 출력)
- ❌ 비교 판정에 LLM·외부 AI 호출
- ❌ 설정값을 코드에 하드코딩 (인코딩, 경로, OS, DB 접속 등)
- ❌ "일단 돌아가게만" 하는 임시 코드를 문서화 없이 남기는 것 (`# TODO:`라도 붙일 것)
- ❌ 한 PR/커밋에 여러 Task를 섞는 것
- ❌ 무거운 프레임워크 도입 (Django, FastAPI 풀스택 등). 프로토 단계임.
- ❌ 데이터 흐름의 시작/끝에 외부 네트워크 호출 (오프라인 동작 보장)

## 7. 자주 헷갈리는 도메인 용어

- **As-Is / 現行**: 메인프레임 원본 환경. 정답지의 출처.
- **To-Be / 新環境**: 리눅스 마이그레이션 후 환경. 검증 대상.
- **현·신 비교 (現新比較)**: As-Is 출력과 To-Be 출력의 동등성 비교.
- **셸 (Shell)**: 여기서는 *Unix shell*이 아니라 **메인프레임 배치 작업 단위**. 1~100개 단위로 검증.
- **CSV**: 변환 툴이 만든 Shift-JIS 인코딩 CSV가 본 도구의 입력.
- **stub 배치**: 프로토 단계에서 진짜 Net COBOL 배치 대신 쓰는 가짜 배치. 인수인계 후 교체 대상.

## 8. 시연 시나리오 (잊지 말 것)

이 프로토는 결국 시연용이다. 시연에서 보여야 할 것:

1. CLI 한 줄로 실행 시작
2. 1~10번 셸이 자동으로 차례차례 처리되는 진행 상황 (적재 → 배치 → 비교 → 결과)
3. 일부는 OK, 일부는 NG (의도적으로 NG 케이스 포함)
4. NG가 난 경우 *어디가 어떻게 다른지*가 터미널에 즉시 표시
5. 끝나면 요약(총 N건 / OK / NG)과 CSV 리포트 경로

이 시나리오가 매끄럽게 굴러가는 것이 **유일한 합격선**이다. 더 화려할 필요 없음.

## 9. 의문이 생기면

- 도메인/사업: `CONTEXT.md`
- 기술 구조: `ARCHITECTURE.md`
- 무엇을 만들어야 하나: `SPEC.md`
- 다음에 뭘 해야 하나: `TASKS.md`
- 왜 이렇게 했나: `DECISIONS.md`
- 위 어디에도 답이 없으면: **사용자에게 질문**. 추측 금지.
