# CONTEXT.md

## 1. 한 줄 요약

일본 기업의 **메인프레임 → 리눅스 마이그레이션** 후, "마이그레이션된 배치가 원본과 같은 결과를 내는지" 자동 검증하는 도구.

---

## 2. 누가 만들고 누가 쓰나

### 2-1. 만드는 사람 (현 단계)

- 외주 개발자 1인 (PM/풀스택 백그라운드)
- 납품처는 **개발자의 아버지가 운영하는 회사** (단순 외주 1회로 끝나지 않을 가능성 큼)

### 2-2. 인수받는 사람

- 아버지 회사의 정직원
- 본인이 만든 Core 코드 위에 GUI를 추가하고, 실 고객사 환경에 설치·운영하게 됨

### 2-3. 최종 사용자 (이 도구를 도입할 고객)

- 일본 중소기업 (메인프레임을 운영하다가 마이그레이션 중인 곳)
- 검증 작업을 **수작업**으로 하거나 **고가 상용 도구**에 비싼 비용을 지불 중
- 그 사이의 **빈틈**을 메우는 것이 본 도구의 시장 위치

---

## 3. 왜 이 도구가 필요한가

### 3-1. 검증의 본질

마이그레이션의 약속은 *"동작은 그대로, 인프라만 바꾼다"*. 이 약속을 검증할 유일한 객관적 방법은 **같은 입력에 같은 출력이 나오는지** 보는 것.

### 3-2. 수작업의 한계

- 셸(배치) 단위가 보통 수십~수백 개
- 사람이 매번 두 출력을 눈으로 대조하는 건 비현실적
- 검증 누락 → 운영 사고의 원인

### 3-3. 상용 도구의 한계

- 일본 메인프레임 마이그레이션용 검증 도구는 존재하지만 비싸다
- 중소기업이 부담하기 어려움
- 기능이 과하거나 사용성이 떨어지는 경우도

### 3-4. 우리의 포지션

- **수작업 < 우리 도구 < 고가 상용 도구**
- 가격은 합리적, 기능은 필수적인 자동화에 집중
- 차별화 포인트: **NG의 원인을 알기 쉽게 분류·설명하는 리포트** (장기 목표)

---

## 4. 사업 모델

### 4-1. 우리 회사의 역할

**비교검증팀** (마이그레이션 자체는 별도 팀/SI가 수행).

- 마이그레이션 SI가 작업을 끝낸 뒤
- 우리가 검증 도구를 **고객의 To-Be 환경에 설치**
- 도구는 고객 환경 안에서 동작 → 데이터 외부 반출 없음

### 4-2. 비즈니스 모델

- **설치형 소프트웨어 라이선스 판매**
- SaaS 아님 (일본 엔터프라이즈 정서상 데이터 반출 꺼림)
- 고객 환경 안에서 닫힌 채로 동작

### 4-3. 도구가 다루는 데이터의 흐름

1. 고객(또는 마이그레이션 팀)이 **As-Is 입력 CSV + As-Is 출력 CSV** 두 종을 메일/USB로 전달
   - 이미 **외부 변환 툴**로 EBCDIC → Shift-JIS CSV 변환이 끝난 상태
2. 검증팀이 받아서 도구에 세팅
3. 도구가 As-Is 입력을 **To-Be의 PostgreSQL에 적재**
4. **To-Be 배치** 실행 (이미 마이그레이션된 Net COBOL 배치)
5. To-Be 출력 CSV 생성
6. **As-Is 출력 vs To-Be 출력** 통짜 바이트 비교
7. 리포트로 결과 제공

## 5. 도메인 환경 (참고 - 실 운영용)

### 5-1. 현행 (As-Is) — 후지쯔 메인프레임

| 항목 | 내용 |
|---|---|
| OS | MSP (후지쯔 메인프레임 OS) |
| 트랜잭션 모니터 | AIM/DC |
| 언어 | COBOL85 (BAGLES II에서 자동 생성) |
| DB | AIM/NDB (네트워크형 DB) |
| 인코딩 | EBCDIC / JEF |

### 5-2. 마이그레이션 후 (To-Be) — 오픈 플랫폼

| 항목 | 내용 |
|---|---|
| 기반 | PITON (확장 미들웨어) |
| 언어 | Net COBOL |
| DB | PostgreSQL (RDB화) |
| 인코딩 | Shift-JIS (변환 후) |

### 5-3. 중요한 함의

- **DB 모델 자체가 다름** (네트워크형 → 관계형). 이게 *정상적으로도* 출력이 미묘하게 달라질 수 있는 가장 큰 원천. **가짜 NG 다발 예상**.
- **BAGLES II 자동 생성 코드**: 패턴이 일관 → NG도 패턴화될 가능성 큼 → 분류 자동화에 유리.
- **인코딩**: 양쪽 Shift-JIS로 통일된 상태에서 비교. 외부 변환 툴이 As-Is를 Shift-JIS로 변환해 준 결과부터가 본 도구의 시작점.

## 6. 본 도구 (프로토 단계)의 범위

### 6-1. 프로토에서 만드는 것

- 우분투 환경에서 PostgreSQL을 깔고
- **stub 배치** (진짜 Net COBOL 배치를 대신할 가짜)와 함께
- 샘플 As-Is CSV 1~10쌍을 처리
- 적재 → 배치 → 비교 → 리포트의 **E2E** 흐름이 자동으로 도는 것을 시연
- 인터페이스는 **CLI** (단, Core 로직은 GUI 확장 가능하게 분리)

### 6-2. 프로토에서 만들지 않는 것

- 진짜 PITON 환경 통합 (인수인계 후 정직원이 교체)
- 진짜 Net COBOL 배치 실행 (stub으로 대체)
- GUI (본인이 별도 단계에서 추가 예정)
- 고객 환경 설치 자동화 (정직원 영역)
- 다양한 클라이언트별 환경 변종 (1개 환경에서만 시연)

### 6-3. 시연 합격 기준

- 샘플 1~10개 CSV를 자동으로 처리
- 일부 NG 케이스를 의도적으로 섞어둠
- NG의 *어느 줄이 어떻게 다른지*가 터미널에 표시
- 끝나면 OK/NG 카운트와 CSV 리포트 경로 출력

---

## 7. 차별화 포인트 (기억해 둘 것)

본 도구의 진짜 가치는 단순 비교가 아니라 **"NG를 알기 쉽게 설명해주는 것"**.

- 데이터가 정말 틀린 NG (마이그레이션 버그)
- 인코딩 불일치로 인한 *가짜* NG
- DB 모델 차이로 인한 *가짜* NG (숫자 포맷, NULL, 공백 등)
- 외자/특수문자 변환으로 인한 *가짜* NG

이 분류를 자동으로 해주는 것이 장기적 차별화. 단, **프로토에선 단계 2까지만** (어느 줄이 다른지 + 그 줄 내용).
