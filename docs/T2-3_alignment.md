## 1. 배경 — 이 변경이 왜 발생했는가

사장님이 별도로 작성한 사업 기획서를 검토한 결과, 우리가 며칠간 합의한 설계와 다음 4가지에서 차이가 있었다.

| 항목 | 사장님 그림 | 현재 우리 합의 | 처리 |
|---|---|---|---|
| 테스트 케이스 수 | 100건 | 10건 (프로토) | 변경 없음 (10건 합의 유지) |
| 정의 파일 (메타데이터) | YAML로 셸별 정보 정의 | 코드에 박힘 | **반영** |
| 입력 데이터 종류 | DB 입력 + 파일 입력 | DB 입력만 | **반영** (야간 배치 흐름으로) |
| 비교 방식 | DB 비교 + 파일 비교 | 파일 비교만 | 변경 없음 (사실상 동일) |

핵심 추가 맥락:
- 사장님과의 대화에서 **야간 배치(夜間バッチ)가 핵심으로 나옴**. 파일 입력 케이스는 단순한 옵션이 아니라 *실제 비즈니스에서 자주 발생하는 워크로드*임.
- 시연용 도메인을 우리가 만들지만, **실 운영에서는 고객의 진짜 스키마를 따르면 됨**. 즉 도구의 진짜 가치는 *동적 적응 능력(정의 파일로 스키마를 받아 처리)*이지, *우리가 도메인을 많이 만드는 것*이 아님.

## 2. 확정된 변경 사항

### 2-1. 정의 파일 도입 (`test_definition.yaml`)

셸별 메타데이터를 코드에서 분리해 정의 파일로 관리한다.

**경량 스키마 (제안):**

```yaml
shells:
  - id: "001"
    input_type: database          # "database" | "file"
    input_table: transaction_log  # input_type == database 일 때 사용
    input_csv: 001.csv
    expected_output_csv: 001.csv
    batch: stub_db                # 어떤 stub 배치를 호출할지

  - id: "006"
    input_type: file              # 야간 배치 시뮬
    input_file_dest: ./tobe_input/  # input_type == file 일 때 사용
    input_csv: 006.csv
    expected_output_csv: 006.csv
    batch: stub_file
```

**스키마 설계 원칙:**
- 사장님 기획서 7.1절의 정신을 따르되 *프로토 단계에 맞는 최소 형태*
- 모든 셸이 같은 메타를 갖지 않을 수 있어야 함 (셸마다 다른 input_type 등)
- 향후 확장 여지 (comparison_rules, success_criteria 등)는 *자리만 비워둬도 OK*

### 2-2. 두 가지 입력 흐름 (DB 입력 + 파일 입력 둘 다 구현)

**DB 입력 흐름 (셸 001~005):**
- 기존 합의 그대로
- As-Is 입력 CSV → PostgreSQL 테이블 적재 → stub이 SELECT → 출력 CSV
- Loader는 기존 `load_input_csv()` 사용

**파일 입력 흐름 (셸 006~010) — 야간 배치 시뮬:**
- As-Is 입력 CSV를 *지정 디렉토리에 복사*
- stub이 *그 파일을 직접 read* + *DB 마스터 SELECT (선택적)* → 출력 CSV
- 새 Loader 함수 필요: `copy_input_file(csv_path, dest_dir)`
- 야간 배치의 전형적 흐름 시뮬: 전날 raw 파일 → 마스터 조인 → 결과 파일

### 2-3. 시연 도메인은 2개로 최소화

- **도메인 A — 결제 (DB 입력, 셸 001~005)**: 기존 `customer_master` + `transaction_log` 활용
- **도메인 B — 야간 배치 시뮬 (파일 입력, 셸 006~010)**: 전날 거래 raw → 마스터 조인 → 일일 명세 파일

**인사·근무기록 등 추가 도메인은 만들지 않음.** 2개로 *동적 적응 능력 + 입력 방식 다양성* 둘 다 시연 가능.

### 2-4. NG/ERROR 케이스 배치

기존 합의 유지하되 *야간 배치 흐름에 배치*:

- 셸 001~005: 모두 OK (결제 도메인, DB 입력)
- 셸 006: OK (야간 배치 정상)
- 셸 007: NG — 한 줄 데이터 차이 (값 1자리 변경)
- 셸 008: NG — 전각 공백 삽입 (顧客名)
- 셸 009: NG — 여러 줄 차이
- 셸 010: ERROR — stub 종료 코드 1 (실행 실패)

야간 배치 쪽에 NG/ERROR가 모이는 게 *시연 임팩트* 면에서 더 강력 (사장님이 핵심으로 언급한 부분에서 검출이 일어남).

### 2-5. 비교 방식은 변경 없음

- 통짜 바이트 비교 (D-004) 유지
- 사장님 기획서의 "DB 비교"는 우리의 "CSV export 후 파일 비교"와 사실상 동일
- HANDOFF.md에 한 줄 명시: *"DB 결과 비교는 CSV export 패턴으로 일관 처리. 사장님 기획서 4.1절의 DB 비교와 사실상 동일 (export 단계가 명시화된 형태)."*

## 4. T2-3 재설계 요청

T2-3 stub 배치를 다음 구조로 재설계할 것:

### 4-1. Stub 배치 종류 2개

- **`stub_batch/run_batch_db.py`** — DB 입력 흐름용 (기존 계획)
- **`stub_batch/run_batch_file.py`** — 파일 입력 흐름용 (신규)

### 4-2. 공통 인터페이스 유지

```bash
python stub_batch/run_batch_{db|file}.py \
  --shell-id 001 \
  --output-path ./out/tobe_output/001.csv \
  --encoding shift_jis \
  --db-host ... --db-port ... --db-name ... --db-user ...
  # DB 입력: --input-table transaction_log
  # 파일 입력: --input-file ./tobe_input/006.csv
  # --clean: NG 주입을 끈 정상 출력 (골든 생성용)
```

### 4-3. 코드 최상단 명시

각 stub 파일 상단에 인수인계 시 교체 포인트 주석을 강화한다.

```python
# === 인수인계 시 교체 포인트 ===
# 이 stub은 시연용. 실 운영에서는 고객의 진짜 배치(Net COBOL)로 교체.
# 입출력 계약 (--shell-id, --output-path, 입력 종류) 유지.
# 본 stub은 도구가 두 입력 흐름(DB / 파일)을 모두 지원함을 시연하기 위한 placeholder.
```

### 4-4. NG 주입 로직

순수 함수 `_apply_ng_pattern(shell_id, rows)`로 분리 (DB 없이 테스트 가능). 셸 007/008/009의 NG 패턴은 *위치 기반·결정론적*.
